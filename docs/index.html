Live-data evaluation

The benchmark your foundation model has not memorised

TSBench-Forge evaluates time-series foundation models using live data, with more weight given to observations generated after model training. An automated discovery system helps the benchmark keep growing as old data becomes stale.

TSBench-Forge · open-source live benchmark for time-series foundation models

Time-series leaderboards look great. That is the problem.

Time-series foundation models have improved rapidly.

Chronos, TimesFM, Moirai, Toto, TiRex and others now report impressive zero-shot results across benchmarks such as GIFT-Eval, BOOM, TIME, Monash and the M competitions.

These are valuable benchmarks, built carefully by serious research teams. But static benchmarks have a structural weakness. Once they are published, they begin to decay.

The data can enter model-training corpora. Nobody needs to cheat for this to happen. Public datasets are naturally swept into increasingly large training mixtures.

A fixed test set also invites optimisation around its particular characteristics. Architecture choices, preprocessing and inference strategies gradually become tuned to the benchmark.

Meanwhile, the world keeps changing. A dataset frozen several years ago measures performance on the dynamics of several years ago.

The result is a credibility gap. Leaderboard scores keep improving, but it becomes harder to know whether the models are improving with them.

This matters beyond academic benchmarking. Developers need to know whether an apparent model improvement will hold up on new data. Businesses choosing between models need evidence that does not come only from vendors. Operators need to know whether a model remains useful as markets, infrastructure and human behaviour change.

As foundation models become more capable and more expensive to deploy, trustworthy evaluation becomes infrastructure.

Keep the data ahead of the models

TSBench-Forge evaluates forecasting models on real data that continues to arrive.

The benchmark collects time series from energy grids, river gauges, earthquake feeds, financial markets, transport systems, health services, environmental sensors and internet infrastructure.

Each source is polled on its own schedule by deterministic scrapers. The benchmark then constructs forecasting challenges from the resulting archive. A model receives 256 recent observations from a real series and must forecast the next 48.

Forecasts are scored using standard point and probabilistic measures, including MASE and CRPS. Each result is expressed relative to seasonal naive, which predicts that the next value will resemble the value from the same point in the previous cycle.

A score below 1.0 means the model beat that baseline.

Every model receives the same seeded challenge set. Comparisons are paired across the same examples, so a small or inconsistent advantage does not automatically become a leaderboard claim.

The benchmark follows four main principles.

1. Give more weight to fresh data

A large share of challenges is sampled from the newest end of each series.

Each challenge is tagged according to how much of its forecast target falls after the relevant training cutoff. Newer answers have more influence on the final result.

Historical windows still provide useful breadth, but their weight is reduced. They cannot dominate the ranking simply because more historical data is available.

The next scoring release will introduce a hard freshness barrier. Forecast targets older than the newest model cutoff used in an evaluation will be rejected before scoring.

The current system strongly rewards freshness and reduces the influence of older observations, but it does not yet eliminate historical exposure completely. We think that distinction is worth stating clearly.

2. Make memorisation difficult and visible

Every served window receives light, truth-preserving changes such as smooth rescaling, time stretching or small patches of hidden history.

The continuation remains real, but the input is no longer byte-identical to a series that may exist in a training corpus. Exact-match lookup becomes much less useful.

TSBench-Forge also includes a behavioural memorisation probe. It checks whether a model performs suspiciously better on previously served windows than on genuinely fresh ones.

No single defence solves contamination. The aim is to combine several protections and measure the remaining risk rather than ignore it.

3. Reject bad data automatically

Not every live feed creates a useful forecasting problem.

Sensors can become stuck. APIs can return almost constant values. Some series are effectively random. Others are so predictable that seasonal naive already solves them.

Every proposed source must pass an automatic quality gate. It checks for degenerate values, flatlines, broken timestamps, excessive duplication and series that appear either unforecastable or trivial.

During a recent source expansion, the system rejected Coinbase’s funding-rate feed because the retrieved series was heavily clamped and repeated. A more informative Kraken feed passed.

A source does not enter the benchmark simply because it sounds interesting.

4. Use baselines that are hard to beat

Seasonal naive is only one comparison.

TSBench-Forge also evaluates exponential moving averages, drift, a simple autoregression and a “context parrot” that copies patterns from recent history.

A foundation model should not rank highly if it cannot separate itself from those approaches.

A live benchmark still needs maintenance

Live data solves part of the staleness problem, but it creates a new one.

Feeds disappear. APIs change. Sites add bot protection. Sensors break. Useful series can become trivial. Coverage gaps emerge as the catalogue changes.

A fixed catalogue of live sources eventually becomes stale too.

Manual curation does not scale well enough to solve this. Finding a promising source, checking its schema, testing it from the real scrape environment, assessing its history and wiring it into the pipeline takes time.

TSBench-Forge includes an LLM-driven discovery system called Autoresearch to reduce that bottleneck.

Its role is deliberately limited.

The agent reads the existing catalogue, identifies weak coverage by domain and update frequency, and proposes candidate sources.

Deterministic code then checks the proposal schema, compares it against a denylist of known training datasets, detects duplicates, probes the endpoint and runs the same quality gate applied to existing sources.

The agent proposes candidates. Code decides whether the data passes. A human approves the final change.

The LLM never touches a benchmark score.

The discovery ledger

Every proposed source is recorded in a persistent ledger, whether it was accepted, rejected, already known, inaccessible or dependent on an API key.

That history is included in future discovery runs.

Early searches proposed Steam player counts dozens of times. Once the ledger became part of the agent’s context, those repeated suggestions disappeared.

The search moved further into the long tail, including unusual government feeds, environmental sensors, infrastructure telemetry and specialised APIs that are less likely to appear in standard machine-learning corpora.

This is where the process begins to compound.

The catalogue grows, but the system also learns where it has already searched, what failed, why it failed and which areas remain underrepresented.

In one recent sweep:

  • 100 discovery rounds generated 1,295 proposals.
  • Those became 663 unique candidates.
  • 454 passed the metadata checks.
  • 273 unique hosts were probed from the actual scrape environment.
  • 118 were confirmed live without requiring an API key.
  • Nine sources were investigated end to end and integrated.

The additions included Gemini trade ticks, NOAA coral reef temperatures, RIPE Atlas root-server latency, US Army Corps reservoir gauges, Swiss lake sensors and Singapore’s live taxi count.

Where APIs exposed enough history, the run also recovered roughly 845,000 rows of backfill.

The search itself cost cents. Human attention was reserved for the small number of candidates that survived the automated stages.

Auditing the benchmark itself

The same quality system used to inspect external sources has also exposed failures inside TSBench-Forge.

During the latest expansion work, the gates identified:

  • a CSV parser silently dropping data from two sources;
  • a deduplication step collapsing separate series that shared timestamps;
  • malformed URLs bypassing duplicate detection.

Each problem was caught before the affected data could influence a model score.

This matters for the economics of maintaining the benchmark. Most bad candidates are rejected automatically, with the reason recorded. Engineers can focus on the shortlist instead of inspecting every proposal manually.

The gates also pressure-test the ingestion pipeline as the catalogue grows.

The automation is not only helping the benchmark expand. It is also helping us find faults in our own system.

Does the live data resemble established benchmarks?

A live benchmark would not be useful if its series represented completely different forecasting problems from those used in established research.

We compared TSBench-Forge with GIFT-Eval, TIME and BOOM using statistical characteristics such as autocorrelation, trend, seasonality, volatility and spectral structure.

The analysis found that TSBench-Forge occupies the same heavily populated regions of forecasting character space as the established suites.

Its centre was especially close to GIFT-Eval’s, and the typical distance from a reference series to its nearest TSBench-Forge neighbour was small.

In practical terms, TSBench-Forge is testing familiar kinds of forecasting problems using data that continues to change.

The comparison also showed where the catalogue lacks coverage. BOOM contains a wider range of observability and infrastructure telemetry. The new RIPE Atlas latency source is an early step towards closing that gap.

Results from the expanded benchmark

We evaluated nine time-series foundation models on the same set of 256 seeded challenges, drawn from 43 series across 62 eligible sources.

Eleven per cent of the challenges came from newly added sources.

Lower scores are better. A relative score of 1.0 means the model performed no better than seasonal naive.

Several results stood out.

Probabilistic forecasting held up on live data

All nine foundation models beat seasonal naive on probabilistic accuracy with strong statistical significance.

The useful result is not simply that the models performed well. Their advantage remained visible when the evaluation shifted towards continually arriving observations.

The strongest advantage was in uncertainty estimates

The models performed consistently well on probabilistic forecasts.

The point-forecasting result was more mixed. Only the strongest models clearly beat the best tuned classical baselines on MASE.

For users who need a single predicted value, a well-chosen moving average remains surprisingly competitive. The larger advantage of foundation models may be their ability to represent a range of possible outcomes.

Harder sources changed the ranking

Adding irregular trade ticks, latency streams with failure spikes and reservoir series with operational guide curves increased error across the board.

The middle of the leaderboard changed. Moirai-2 moved up three places, while Toto-2 and TabPFN-TS remained at the top.

That movement is part of what a live benchmark should expose.

A static benchmark produces a ranking for one fixed dataset. A live benchmark can show which models remain reliable as the data changes.

The exact ordering from one run matters less than whether the results hold up next month.

Current limitations

TSBench-Forge is still early.

Breadth remains its clearest weakness. The characterised pool contains hundreds of series rather than hundreds of thousands, and this run drew challenges from 43 of them.

Challenges from the same source may also be correlated. The statistical comparisons are paired per challenge rather than treating every example as completely independent.

Freshness is currently weighted rather than absolute. Historical windows are perturbed, down-weighted and audited, but part of the score may still come from observations a model could theoretically have encountered.

The hard post-cutoff barrier is intended to close that gap in the next scoring release.

Live sources also have lifecycles. Some begin with deep public histories. Others only accumulate data after the scraper is activated. Seasonal feeds can remain silent for months. An Iowa snow-plough source, for example, is wired but has to wait for winter before it becomes useful.

A live benchmark is also not reproducible in exactly the same way as a frozen archive.

Given the same data pool and seed, TSBench-Forge produces the same challenge set. The pool itself, however, continues to grow.

We see that as a worthwhile trade-off.

What comes next

The next scoring release will enforce the hard freshness barrier, making post-cutoff targets a property of the benchmark rather than a weighting preference.

We also plan to run the leaderboard regularly using newly arrived challenges and publish changes in model performance over time.

The discovery system will continue expanding the catalogue, with an initial focus on observability and infrastructure telemetry.

The public results will also include richer measures of calibration and interval coverage.

Forecast quality is not only about predicting the centre correctly. When a forecast informs a real decision, users need to know whether its uncertainty estimates can be trusted.

The longer-term goal is independent validation. Challenge seeds could be committed before the future observations they select exist, with scores reproduced by parties other than us.

The ideal benchmark is one whose results do not depend on trusting the people who built it.

Takeaway

Static benchmarks ask which model performed best on the published past.

TSBench-Forge asks which model forecasts new observations best as data arrives and real systems change.

The full pipeline, including the scrapers, quality gates, source-discovery system, proposal ledger and statistical evaluation harness, is open source in the TSBench-Forge repository.

To propose a source, open an issue with the feed URL and update frequency, or run the vetting tools against your own candidate list.

The same deterministic gates used for the discovery agent’s proposals will evaluate yours.
